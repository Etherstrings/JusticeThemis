## Context

当前项目的主链路是“抓新闻 + 抓市场数据 + 生成日报 + 导出 handoff/prompt/MMU + 多种展示出口”，但产品层实际上被 free/premium 分层和旧日报产物绑住了。结果是：

- 采集、分析、导出和展示边界不清
- 结论和证据经常分离，用户看到的是结果，不知道结果怎么来的
- 导出层围绕 tier 组织，而不是围绕“这段时差里世界发生了什么”
- 缺少一个能被直接转发、截图、打开就能看懂的核心结果图

这次重构不是补一个新展示层，而是把项目重新收成一个单一产品：前面是来源模块，后面是分析对照模块，最终输出“昨天世界钱往哪里”。

## Goals / Non-Goals

**Goals:**

- 把现有服务拆成两个清晰平面：来源采集平面、分析对照平面
- 移除 free/premium 双版本分析产物，统一成单一日报和单一导出物
- 在不删除任何现有网站、信源和采集逻辑的前提下，定义一个稳定的“昨天世界钱往哪里”结构化载荷
- 支持两条成图路径：内部成图，或直接导出成图数据给外部 AI / 本地模板项目使用
- 保留管理操作鉴权，但去掉 premium 只读内容鉴权和相关运行时依赖

**Non-Goals:**

- 不在这次 change 中重做所有新闻打分和主题分类规则
- 不在这次 change 中引入用户体系、订阅体系或支付能力
- 不在这次 change 中扩成通用新闻平台或多租户服务
- 不要求一次性删除所有历史导出文件，只要求新主链路不再依赖 tier 分叉
- 不以“精简信源”为目标，不在这次 change 中删除现有来源网站或已有抓取逻辑

## Decisions

### Decision: 用两个顶层产品模块重构服务边界

新结构以“来源模块”和“分析对照模块”为中心，而不是以日报种类或导出类型为中心。

- 来源模块负责：
  - 新闻源抓取
  - 全球权益、行业板块、利率、汇率、商品、宏观/政策、地缘、AI/科技和大公司事件抓取
  - 来源状态、刷新状态、可用性和标准化输出
  - 在来源层完成分层、筛选、去噪和简短解释，而不是把生数据整包丢给后面
  - 保留现有来源注册表和已有抓取逻辑，只做重组、抽象和标准化
- 分析对照模块负责：
  - 从来源模块读取标准化输入
  - 生成主结论、资金流判断、跨资产关联和不确定点
  - 保留结论到输入的可回看关系
  - 输出“昨天世界钱往哪里”成图数据

这样能让后续开发时知道“这个功能是在拿数据、整理数据，还是在解释数据”。

Alternatives considered:

- 继续按现有 service 文件逐步补逻辑：短期改动小，但会继续把产品边界搅在一起。
- 先只改导出文案：看起来会更像产品，但底层产物还是 tier 化和混线结构，后面还得再拆一次。

### Decision: 取消 free/premium，统一成单一分析产物

现有 free/premium 结构把分析、prompt、MMU、鉴权和导出都分叉了，但用户现在要的是一个能直接看的产品，不是两套内容。重构后保留单一分析产物：

- 单一日报缓存
- 单一方向判断
- 单一资产组/主题观察层

需要管理权限的仍然是刷新、生成、配置等操作；只读内容不再按 premium 保护。

Alternatives considered:

- 保留免费/付费，只是不在最终图卡里展示分叉：看起来简单，但后端复杂度、配置复杂度和导出复杂度都还在。
- 改成更多层级：只会让产品更虚，不会更清楚。

### Decision: 以“昨天世界钱往哪里”成图数据为核心，再决定是否内部出图

“昨天世界钱往哪里”不能直接从零拼图，而要先有一个稳定的结构化 payload。这里优先回答的是“这段时差里世界上发生了什么，钱主要往哪边走了”，不是“今天具体该买什么”。新的输出分三层：

1. `source_intel_payload`
   - 已经做过第一轮编辑的来源层输入
   - 包含重要度、分组、去重、简短解释、来源和时间
2. `yesterday_world_money_flow_payload`
   - 直接给图片使用的全球隔夜总览数据
   - 固定分区为：
     - `time_window`
     - `global_headline`
     - `headline_events`
     - `macro_policy`
     - `geopolitics`
     - `equity_and_sector_moves`
     - `rates_and_fx`
     - `commodities`
     - `tech_ai_and_major_companies`
     - `cross_asset_links`
     - `source_index`
   - 每条条目至少包含：
     - `title`
     - `summary`
     - `why_it_matters`
     - `importance_tier`
     - `topic`
     - `direction`（如适用）
     - `linked_assets`
     - `published_at`
     - `source`
     - `group_id`
3. 渲染层
   - 如果本地有稳定模板项目，例如 `stockComImage`，则优先适配模板输入
   - 如果本地模板不可用，则先输出可直接喂给 AI 成图的数据载荷
   - 内部 SVG/PNG 渲染器作为可选增强，而不是首个阻塞点

先有 payload，图片和分享卡片都能共用一套数据；这样就算内部成图暂时不做，也能立刻把数据交给外部 AI 或模板项目。

### Decision: 来源模块不是原始仓库，而是第一层情报编辑面

来源模块不应该只是“抓回来、存进去”。第一部分本身就要完成第一轮编辑工作，至少包括：

- 重要度分层
- 主题归类
- 噪音过滤
- 重复项压缩
- 每条输入的一句人话解释

这样第二部分做对照和映射时，面对的是“已经能读的情报输入”，不是一坨裸数据。

Alternatives considered:

- 来源层只做抓取，所有解释都压到第二部分：会让第二部分又脏又重，而且第一层输出会显得很傻。
- 来源层直接做完整结论：会和第二部分的职责混掉。
- 直接人工截图导图：实现快，但服务端和导出链路无法稳定复用。
- 完全依赖内部 SVG 渲染：如果模板风格未定，会拖慢主链路落地。

### Decision: 分析对照模块必须保留“结论 -> 输入”的回链

重构后的主结果不再只是 summary 文本。每条重点结论都必须能回看：

- 对应哪几条新闻
- 对应哪些市场观察
- 方向是利多还是利空
- 为什么不是高置信度

这会形成一个统一结构，比如：

- `headline`
- `flow_direction`
- `drivers`
- `linked_items`
- `linked_market_observations`
- `uncertainty_note`
- `confidence`

这样导图和成图数据都能说人话，但不会变成“只有一句判断，没有来由”。

Alternatives considered:

- 继续生成长段落叙述：读起来完整，但不利于卡片化成图，也不利于用户快速扫。
- 只保留 tags 和 score：机器味太重，不像产品。

### Decision: 管理权限保留，premium 运行时配置删除

重构后权限模型收敛成两类：

- 管理操作：仍需 admin key 或 unsafe dev override
- 读取内容：公开可读，不再要求 premium key

这意味着：

- 去掉 `OVERNIGHT_PREMIUM_API_KEY` 相关运行时要求
- 去掉 premium tier 路由依赖
- 保留 admin 保护的刷新、生成、配置和 readiness

Alternatives considered:

- 所有内容都放开且管理操作也不保护：不适合现有本地/操作员工作流。
- 继续保留 premium key 但只在少数导出链路里用：增加理解成本，没有实际产品收益。

## Risks / Trade-offs

- [重构跨度大，旧导出链路很多] → 先定义单一主链路和兼容期，旧文件先保留但不再作为主产品依赖。
- [去掉 tier 后会影响现有 prompt/MMU 逻辑] → 在 spec 中明确迁移到单一分析产物，优先让 prompt/MMU 读取新统一结构。
- [“昨天世界钱往哪里”如果字段太多，会又变回信息堆料] → 固定图中分区和每区卡片数量，优先回答“发生了什么”“钱往哪走”“哪里还不确定”。
- [重构时误删已有信源或采集逻辑] → 先做来源资产清单，要求迁移前后来源注册表和抓取入口一一对齐。
- [第一部分如果不做筛选解释，会重新变成原始数据垃圾堆] → 在来源模块里强制加入分层、去重、简短解释和噪音过滤输出。
- [来源模块与分析模块拆分不彻底，会形成名义拆分] → 以独立 payload 和接口边界约束，不允许分析模块直接吃原始 provider payload。
- [图卡风格和最终成图工具未定] → 先把 render-ready payload 做稳定，再补模板适配或内部渲染。

## Migration Plan

1. 定义新的来源模块输出结构，统一新闻项、市场观察、来源状态和刷新状态，并加入分层筛选、去重压缩与简短解释字段。
2. 定义新的分析对照结构和单一日报缓存，停止生成 free/premium 分叉结果。
3. 增加 `yesterday_world_money_flow_payload`，固定全球分区，优先产出成图数据，再按本地条件接模板项目或内部导图。
4. 调整 API 和导出命令，使其消费新的单一分析结构和成图数据结构。
5. 把 prompt/MMU/export 迁移到新的单一分析产物。
6. 清理 premium key 依赖、tier 路由和相关文档。
7. 保留兼容窗口，允许旧历史产物继续存在，但新生成链路不再写 tier 分叉文件，也不删现有信源定义。

## Open Questions

- 首版是否只输出成图数据 JSON/Markdown，还是同时补一个内部 SVG 版本。
- 第一版是否先把 `stockComImage` 作为适配目标，还是先保持独立的 render-ready payload。
- 当新闻信号和市场信号冲突时，第一张图默认是以“新闻发生了什么”为主，还是以“价格怎么走”为主。
